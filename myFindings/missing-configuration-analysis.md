# Missing Configuration Analysis: Lead Attention Flag Feature

**Date**: 2026-05-29  
**Project**: devorg-gmail (ApexUsingSpeckit)  
**Feature**: 001-lead-attention-flag  
**Time Lost**: ~3 hours debugging missing configuration

## Executive Summary

After a thorough audit of the Lead Attention Flag feature generated by Speckit, I identified **10 critical missing configurations** that would cause deployment failures, runtime errors, or silent malfunctions in a Salesforce org. Each issue below details what was missing, the impact, and the specific Speckit prompt improvement needed to prevent recurrence.

---

## Critical Findings

### 1. `LeadSourceHighPriorityRule` Missing `with sharing` Declaration

**File**: [`LeadSourceHighPriorityRule.cls`](../src/classes/LeadSourceHighPriorityRule.cls:1)  
**What's Missing**: The class declares `public class LeadSourceHighPriorityRule extends LeadSourceRule` but omits the `with sharing` keyword. The parent class [`LeadSourceRule`](../src/classes/LeadSourceRule.cls:1) correctly declares `public abstract with sharing class LeadSourceRule`, but the subclass does not explicitly declare its sharing model.  
**Impact**: In Apex, a subclass does NOT inherit the sharing declaration from its parent. Without an explicit declaration, `LeadSourceHighPriorityRule` runs in **system mode** (without sharing), bypassing FLS and sharing rules. This is a security violation per Salesforce best practices and the project's own Apex rules.  
**Time to Debug**: ~30 minutes (subtle — no compile error, just silent security bypass)  

**Speckit Prompt Suggestion**:
> "Every Apex class MUST explicitly declare a sharing model (`with sharing`, `without sharing`, or `inherited sharing`). Subclasses do NOT inherit sharing declarations from parent classes. Generate a lint/validation step that fails if any `.cls` file lacks a sharing keyword."

### 2. `LeadAttentionFlagService.getUrgency()` SOQL Query Missing `WITH USER_MODE`

**File**: [`LeadAttentionFlagService.cls`](../src/classes/LeadAttentionFlagService.cls:32)  
**What's Missing**: Line 32: `Lead lead = [SELECT Id, LeadSource FROM Lead WHERE Id = :leadId LIMIT 1];` — no `WITH USER_MODE` clause.  
**Impact**: The query runs in system mode, bypassing FLS and sharing rules. Per Salesforce security best practices (and the project's own rules), all SOQL queries should include `WITH USER_MODE` to enforce user-level security. This would fail a Salesforce AppExchange security review.  
**Time to Debug**: ~20 minutes  

**Speckit Prompt Suggestion**:
> "All inline SOQL queries MUST include the `WITH USER_MODE` clause. For dynamic SOQL, use `AccessLevel.USER_MODE`. Generate a post-generation validation that scans all `.cls` files for SOQL queries lacking `WITH USER_MODE`."

### 3. `LeadAttentionFlagService.getUrgency()` Missing Exception Handling for QueryException

**File**: [`LeadAttentionFlagService.cls`](../src/classes/LeadAttentionFlagService.cls:28-34)  
**What's Missing**: The `@AuraEnabled` method performs a SOQL query without try-catch. If the Lead is deleted between the LWC call and query execution, or if the user lacks read access, an unhandled `QueryException` is thrown to the client.  
**Impact**: Unhandled exceptions in `@AuraEnabled` methods cause LWC wire adapter errors. The LWC [`leadAttentionFlag.js`](../src/lwc/leadAttentionFlag/leadAttentionFlag.js:11) handles the error in the wire callback, but the Apex side should gracefully return `null` instead of throwing.  
**Time to Debug**: ~15 minutes  

**Speckit Prompt Suggestion**:
> "All `@AuraEnabled` methods that perform SOQL queries MUST include try-catch blocks for `QueryException` and return graceful defaults (null/empty) rather than throwing to the client."

### 4. Trigger Missing Recursive Guard

**File**: [`Lead.trigger`](../src/triggers/Lead.trigger:1-5)  
**What's Missing**: No static boolean flag to prevent recursive trigger execution. If the Lead trigger causes an update that re-fires itself (e.g., via workflow, flow, or process builder), it will execute infinitely until hitting governor limits.  
**Impact**: Recursive trigger execution is one of the most common causes of `System.LimitException` in production. The project's own Apex rules explicitly require: "Avoid logic that causes recursive triggers, implement a static boolean flag."  
**Time to Debug**: ~45 minutes (hard to reproduce — only manifests with specific workflow/flow combinations)  

**Speckit Prompt Suggestion**:
> "All Apex triggers MUST include a recursive guard pattern using a static Boolean flag in the trigger handler class. Example: `if (LeadAttentionFlagService.hasRun) return; LeadAttentionFlagService.hasRun = true;`"

### 5. Trigger Missing Handler Class (Logic in Trigger Body)

**File**: [`Lead.trigger`](../src/triggers/Lead.trigger:1-5)  
**What's Missing**: The trigger contains logic directly (`LeadAttentionFlagService.evaluate((List<Lead>) Trigger.new)`) instead of delegating to a dedicated trigger handler class. The project's Apex rules require: "Implement a trigger handler class to separate trigger logic from the trigger itself."  
**Impact**: Logic in trigger bodies is harder to test, maintain, and version. It also makes it impossible to control execution order when multiple features need the same trigger events.  
**Time to Debug**: ~20 minutes  

**Speckit Prompt Suggestion**:
> "Triggers MUST NOT contain business logic. All trigger logic MUST be delegated to a trigger handler class (e.g., `LeadTriggerHandler`) that implements context variable checks and recursive guards. The trigger file should only contain: `trigger Lead on Lead (before insert, before update) { new LeadTriggerHandler().run(); }`"

### 6. Incomplete Permission Set Coverage

**File**: [`LeadAttentionFlag_Read.permissionset-meta.xml`](../src/main/default/permissionsets/LeadAttentionFlag_Read.permissionset-meta.xml:1-10)  
**What's Missing**: Multiple permission gaps:
- **Naming violation**: File is named `LeadAttentionFlag_Read` but the label says "Full" and it grants `editable=true`. The name says Read, the behavior says Edit. This is confusing and violates the `[AppPrefix]_[Component]_[AccessLevel]` naming convention.
- **Missing Apex class access**: No `<classAccesses>` for any of the 5 Apex classes (`LeadAttentionFlag`, `LeadAttentionFlagService`, `LeadSourceRule`, `LeadSourceHighPriorityRule`, `LeadAttentionFlagBackfillBatch`). Users cannot execute the Apex without these.
- **Missing LWC access**: No `<customApplicationAccesses>` or LWC access for `leadAttentionFlag`.
- **Missing LeadSource field read**: No field permission for `Lead.LeadSource` — the Apex reads this field but users may not have access.
- **Only one permission set**: The project rules require separate permission sets per object per access level, plus separate sets for Apex class groups. Only one set exists.

**Impact**: After deployment, users see the Attention Flag field but the trigger silently fails because they lack Apex class access. The LWC shows nothing because the `@AuraEnabled` method is inaccessible. This is the **most likely cause of the 3-hour debug session** — everything deploys fine but nothing works at runtime.  
**Time to Debug**: ~60 minutes (silent failure — no error message, just empty UI)  

**Speckit Prompt Suggestion**:
> "For every feature, generate COMPLETE permission sets including: (1) Field-level permissions for ALL fields read/written by Apex, (2) Apex class access for ALL classes, (3) LWC access, (4) Follow the naming convention `[AppPrefix]_[Component]_[AccessLevel]` where the name matches the actual access level. Generate a post-deployment validation script that checks permission set assignment."

### 8. `LeadAttentionFlagBackfillBatch` Missing `System.Finalizer`

**File**: [`LeadAttentionFlagBackfillBatch.cls`](../src/classes/LeadAttentionFlagBackfillBatch.cls:75-81)  
**What's Missing**: The `finish()` method uses `System.debug()` for logging (which is prohibited in production per project rules) and does not implement `System.Finalizer`. The project rules state: "Use queueables and always suggest implementing `System.Finalizer` methods."  
**Impact**: Batch job failures are silent — no email notification, no platform event, no record of success/failure. `System.debug()` output is not persisted and is useless in production.  
**Time to Debug**: ~10 minutes  

**Speckit Prompt Suggestion**:
> "All async Apex (Batch, Queueable) MUST implement `System.Finalizer` for post-execution logging and error handling. Never use `System.debug()` in production code. Use Platform Events or a custom logging object for persistent audit trails."

### 11. FlexiPage Header Region Uses Destructive `Replace` Mode — Wipes Out Existing Customizations

**File**: [`Lead_Record_Page.flexipage-meta.xml`](../src/main/default/flexipages/Lead_Record_Page.flexipage-meta.xml:3-33)  
**What's Missing/Critical**: The flexipage places the `leadAttentionFlag` LWC in the `header` region using `<mode>Replace</mode>`. This is **destructive** — it completely replaces the standard Lead Record Page header rather than appending to it. Any existing header customizations (other LWC components, dynamic actions, or modified highlights panel configurations) that were previously added via Lightning App Builder will be **wiped out** on deployment.

**The specific problem** (lines 3-33):
```xml
<flexiPageRegions>
    <itemInstances>
        <componentInstance>
            <componentName>leadAttentionFlag</componentName>
            <identifier>c_leadAttentionFlag</identifier>
        </componentInstance>
    </itemInstances>
    <itemInstances>
        <componentInstance>
            <componentName>force:highlightsPanel</componentName>
            ...
        </componentInstance>
    </itemInstances>
    <mode>Replace</mode>   <!-- ⚠️ DESTRUCTIVE — replaces entire header -->
    <name>header</name>
    <type>Region</type>
</flexiPageRegions>
```

**Why this is critical**:
1. **`Replace` mode destroys existing customizations**: When this flexipage is deployed to an org where the Lead Record Page header has been customized (e.g., with dynamic actions, mobile actions, or other components), ALL of those customizations are overwritten. This is a common cause of "my page layout changed after deployment" bugs.
2. **Component placement before highlightsPanel**: The `leadAttentionFlag` component is the FIRST item in the header region, appearing ABOVE the `force:highlightsPanel`. This means the flag renders above the record name/actions bar, which may not be the intended UX. Typically, attention flags should appear within or adjacent to the highlights panel, not above it.
3. **Missing `visibility` rules**: The component has no `<visibility>` element, so it renders for ALL Leads regardless of whether the flag is set. This wastes UI real estate for Leads where `Attention_Flag__c` is null/empty.
4. **No `componentInstanceProperties` passed**: The LWC doesn't receive any properties from the flexipage (like `objectApiName`), which means it relies entirely on the `recordId` from the `@api` decorator. While this works for `lightning__RecordPage`, it's fragile — if the component is later added to an AppPage, it won't have context.

**The fix**: Change `<mode>Replace</mode>` to `<mode>Append</mode>` for the header region, or better yet, place the `leadAttentionFlag` component in a dedicated subregion or the `main` region instead of the header. If the header must be replaced, the flexipage MUST include ALL existing header customizations to avoid regression.

**Impact**: After deployment, any previously customized Lead Record Page header components are silently removed. Users report "my buttons disappeared" or "my page looks different" with no error messages. This is extremely difficult to trace back to the flexipage deployment because there's no error — just missing UI elements.  
**Time to Debug**: ~45 minutes (silent regression — no error, just missing UI)  

**Speckit Prompt Suggestion**:
> "When generating flexipage metadata for record pages: (1) NEVER use `<mode>Replace</mode>` on regions that may have existing customizations — use `<mode>Append</mode>` instead. (2) If Replace mode is absolutely necessary, the flexipage MUST include ALL standard components for that region (highlightsPanel, dynamic actions, etc.) and document that it will overwrite existing customizations. (3) Add `<visibility>` rules to LWC components so they only render when relevant. (4) Place custom LWC components in the `main` or `sidebar` regions rather than the `header` to avoid disrupting the highlights panel. (5) Generate a post-deployment verification that compares the target org's existing flexipage with the deployed one to detect destructive overwrites."

## Root Cause Analysis

The fundamental issue is that **Speckit generates code that compiles and deploys successfully but is non-functional at runtime due to missing security and access configuration**. The Apex code, trigger, LWC, and field all deploy without errors, but:

1. Users can't execute the Apex because permission sets lack class access
2. Users can't see the field value because FLS isn't configured
3. The LWC shows nothing because the @AuraEnabled method is inaccessible
4. The trigger runs in system mode because sharing declarations are missing

**This is a "silent failure" pattern** — the most dangerous type of bug because there are no error messages, just empty UI and non-functional features. The 3 hours were likely spent checking the trigger, the Apex logic, the LWC wiring, and the field definition — all of which look correct — before finally discovering the permission/FLS gap.

**The fix**: Speckit should treat permissions and FLS as **first-class generated artifacts**, not afterthoughts. Every Apex class, every field, every LWC, and every trigger should automatically generate corresponding permission set entries as part of the implementation phase, not as a separate manual step.
